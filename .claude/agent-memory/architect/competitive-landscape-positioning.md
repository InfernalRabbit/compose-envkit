Finding from a 17-agent adversarially-verified landscape scan (workflow `cenvkit-landscape-scan`, run wf_96edac1c-864, 2026-06-18). Decompose cenvkit's value into 3 capabilities and ask "is each already served?":

1. **Layered-chain DELIVERY to any consumer** (populate+exec): NOT novel — `mise`, `dotenvx`, 1Password `op run`, `chamber`, `sops exec-env`, and Docker Compose's own `--env-file`/`COMPOSE_ENV_FILES` all do it. `cenvkit compose`/`run` is a thin wrapper over a native capability.
2. **Chain DEBUGGING WITHOUT compose** (per-var provenance): NOT novel, and cenvkit loses — `mise` (`mise env --json-extended`, `mise set` source column), **Dynaconf** (`inspect`/`get_history` with `SourceMetadata` — richer than cenvkit), `dotenvx --verbose`. They also support JSON/YAML/TOML env files; cenvkit's chain reader is minimal `KEY=VALUE` only.
3. **Chain DEBUGGING WITH compose** — source-attributed provenance + `env_file:`→`${VAR}` gap-detection: **NOVEL, uncontested.** `docker compose config --environment` is a flat KEY=value dump with zero source attribution and cannot see `env_file:` as an interpolation source; every other tool is compose-blind. Docker closed **#13190 as not-planned**, #3435 open since 2016 → the gap is permanent by design.

**Why it matters:** The "make it work not only with compose / pure env populator / deliver+debug chains without compose" direction the user explored 2026-06-18 marches cenvkit INTO the crowded field (caps 1+2, where mise/Dynaconf win) and AWAY from its only moat (cap 3). "Do one thing well" is the right instinct, but the one thing should be **"the Docker Compose `env_file:` gap debugger/linter,"** not "an env populator."

**How to apply:** Recommend foregrounding `env-debug` gap-detection as THE product; keep `cenvkit compose`/`env-files`/`run` as honest thin convenience over native `COMPOSE_ENV_FILES`. Frame the without-docker mode as "inspect the SAME chain compose will use, pre-flight/CI" (an angle mise/Dynaconf can't match), NOT as a general settings-provenance feature. Propose `cenvkit gap-report`/lint with a non-zero CI exit code as the defensible install reason — and it runs **pre-build** (before `docker build`), daemon-free. Pin compose-go + add an acceptance test asserting gap fidelity against the real loader.

**User rationale + reframe (2026-06-18):** the user justified the generic populator NOT as a market play but to avoid maintaining TWO tools — same chain/values feeding both `docker compose` AND local dev without docker. So populator + gap-debugger **COEXIST (not either/or)**: gap-debugger = the moat (and a pre-build CI lint), populator (`run` + `--expand` + `env --format` emit) = the equal "local arm" of the same chain. Don't market the populator as a mise-killer; do build it for internal consistency.

**cenvkit's concrete populate deficits vs competitors (verified, same scan), ranked by relevance to the "one tool" goal:** (1) no generic `run -- cmd` exec yet — only feeds compose (mise/dotenvx/op run/sops/chamber all do `tool -- cmd`) → BUILD; (2) no `${VAR}`/`${VAR:-default}` value expansion — parser reads literal KEY=VALUE (compose expands for the compose path; nobody expands for local-no-docker) → BUILD as `--expand`; (3) no value emit — `env-files` prints PATHS not KEY=VALUE; no `env --format dotenv|json|shell` → BUILD; (4) no secret masking in output — `env-debug --overview/--value` prints raw secret values (mise redact / op run masks) → cheap MUST-FIX; (5) secrets are plaintext `.secrets.env` only, no encryption/backend (dotenvx/sops/op run/mise) → likely OUT of scope (thin: let an external tool bring secrets + mask them); (6) dotenv-only, no JSON/YAML/TOML env files (mise/Dynaconf) → skip (YAGNI); (7) fewer precedence toggles → fine (predictability is a feature). Positioning decision still OPEN with the user as of 2026-06-18. See also [[layer2-debug-only-not-populated]], [[thin-engine-compose-owns-resolution]].
